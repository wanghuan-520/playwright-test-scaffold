# 任务：[功能名称]

**输入**：来自 `/specs/[###-feature-name]/` 的设计文档
**前提条件**：plan.md（必需）、spec.md（用户故事必需）、research.md、data-model.md、contracts/

**测试**：下面的示例包括测试任务。测试是可选的 - 仅在功能规约中明确要求时才包括它们。

**组织**：任务按用户故事分组，以实现每个故事的独立实现和测试。

## 格式：`[ID] [P?] [Story] 描述`

- **[P]**：可以并行运行（不同文件，无依赖关系）
- **[Story]**：此任务属于哪个用户故事（例如，US1、US2、US3）
- 在描述中包含确切的文件路径

## 路径约定

- **单个项目**：仓库根目录的 `src/`、`tests/`
- **Web 应用**：`backend/src/`、`frontend/src/`
- **移动**：`api/src/`、`ios/src/` 或 `android/src/`
- 下面显示的路径假定单个项目 - 根据 plan.md 结构调整

## 依赖关系和执行顺序

### 阶段依赖关系

- **设置（第 1 阶段）**：无依赖关系 - 可以立即开始
- **基础（第 2 阶段）**：依赖设置完成 - 阻止所有用户故事
- **用户故事（第 3+ 阶段）**：所有依赖基础阶段完成
  - 用户故事然后可以并行进行（如果有人员）
  - 或按优先级顺序顺序进行（P1 → P2 → P3）
- **完善（最终阶段）**：依赖所有所需用户故事完成

### 用户故事依赖关系

- **用户故事 1（P1）**：可以在基础（第 2 阶段）后开始 - 不依赖其他故事
- **用户故事 2（P2）**：可以在基础（第 2 阶段）后开始 - 可能与 US1 集成，但应该可独立测试
- **用户故事 3（P3）**：可以在基础（第 2 阶段）后开始 - 可能与 US1/US2 集成，但应该可独立测试

### 在每个用户故事内

- 测试（如果包括）必须在实现前编写并失败
- 模型在服务之前
- 服务在端点之前
- 核心实现在集成之前
- 故事完成后再移至下一个优先级

### 并行机会

- 所有标记为 [P] 的设置任务可以并行运行
- 所有标记为 [P] 的基础任务可以并行运行（在第 2 阶段内）
- 一旦基础阶段完成，所有用户故事可以并行开始（如果团队容量允许）
- 用户故事的所有标记为 [P] 的测试可以并行运行
- 故事内标记为 [P] 的模型可以并行运行
- 不同的团队成员可以并行处理不同的用户故事

## 实现策略

### MVP 优先（仅用户故事 1）

1. 完成第 1 阶段：设置
2. 完成第 2 阶段：基础（关键 - 阻止所有故事）
3. 完成第 3 阶段：用户故事 1
4. **停止并验证**：独立测试用户故事 1
5. 如果准备好则部署/演示

### 增量交付

1. 完成设置 + 基础 → 基础准备就绪
2. 添加用户故事 1 → 独立测试 → 部署/演示（MVP！）
3. 添加用户故事 2 → 独立测试 → 部署/演示
4. 添加用户故事 3 → 独立测试 → 部署/演示
5. 每个故事在不破坏先前故事的情况下增加价值

### 并行团队策略

有多个开发人员：

1. 团队一起完成设置 + 基础
2. 一旦基础完成：
   - 开发人员 A：用户故事 1
   - 开发人员 B：用户故事 2
   - 开发人员 C：用户故事 3
3. 故事独立完成和集成

## 注意事项

- [P] 任务 = 不同文件，无依赖关系
- [Story] 标签将任务映射到特定用户故事以进行可追溯性
- 每个用户故事应该可独立完成和测试
- 在实现前验证测试失败
- 在每个任务或逻辑组后提交
- 在任何检查点停止以独立验证故事
- 避免：模糊任务、相同文件冲突、破坏独立性的跨故事依赖关系
