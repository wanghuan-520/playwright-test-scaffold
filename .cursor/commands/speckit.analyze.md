## 用户输入

```text
$ARGUMENTS
```

你**必须**在继续之前考虑用户输入（如果非空）。

## 目标

在实现前识别三个核心文档（`spec.md`、`plan.md`、`tasks.md`）之间的不一致、重复、模糊和规约不足的项目。此命令必须仅在 `/speckit.tasks` 成功生成完整的 `tasks.md` 之后运行。

## 操作约束

**严格只读**：不要修改任何文件。输出结构化分析报告。提供可选的修复计划（用户必须明确批准，才能手动调用任何后续编辑命令）。

**宪法权威**：项目宪法（`.specify/memory/constitution.md`）在此分析范围内**不可协商**。宪法冲突自动为 CRITICAL，需要调整规约、计划或任务 - 而不是稀释、重新解释或默默忽略原则。如果原则本身需要更改，则必须在 `/speckit.analyze` 之外的单独、明确的宪法更新中进行。

## 执行步骤

### 1. 初始化分析上下文

从仓库根目录运行一次 `.specify/scripts/bash/check-prerequisites.sh --json --require-tasks --include-tasks` 并解析 JSON 以获取 FEATURE_DIR 和 AVAILABLE_DOCS。派生绝对路径：

- SPEC = FEATURE_DIR/spec.md
- PLAN = FEATURE_DIR/plan.md
- TASKS = FEATURE_DIR/tasks.md

如果任何必需文件缺失，则中止并显示错误消息（指示用户运行缺失的前提条件命令）。
对于参数中的单引号（如 "I'm Groot"），使用转义语法：例如 'I'\''m Groot'（或如果可能使用双引号："I'm Groot"）。

### 2. 加载文档（渐进式披露）

从每个文档中仅加载最少必要的上下文：

**来自 spec.md**：

- 概览/上下文
- 功能需求
- 非功能需求
- 用户故事
- 边缘情况（如果存在）

**来自 plan.md**：

- 架构/技术栈选择
- 数据模型引用
- 阶段
- 技术约束

**来自 tasks.md**：

- 任务 ID
- 描述
- 阶段分组
- 并行标记 [P]
- 引用的文件路径

**来自宪法**：

- 加载 `.specify/memory/constitution.md` 以进行原则验证

### 3. 构建语义模型

创建内部表示（不包括输出中的原始文档）：

- **需求清单**：每个功能 + 非功能需求及稳定键（基于命令短语派生 slug；例如，"用户可以上传文件" → `user-can-upload-file`）
- **用户故事/动作清单**：具有验收标准的离散用户动作
- **任务覆盖映射**：将每个任务映射到一个或多个需求或故事（通过关键字 / 明确引用模式推断，如 ID 或关键短语）
- **宪法规则集**：提取原则名称和 MUST/SHOULD 规范性声明

### 4. 检测通道（Token 高效分析）

专注于高信号发现。限制为总共 50 个发现；在溢出摘要中聚合剩余部分。

#### A. 重复检测

- 识别近似重复需求
- 标记质量较低的措辞以进行合并

#### B. 模糊检测

- 标记缺乏可测量标准的模糊形容词（快速、可扩展、安全、直观、健壮）
- 标记未解决的占位符（TODO、TKTK、???、`<placeholder>` 等）

#### C. 规约不足

- 具有动词但缺少对象或可测量结果的需求
- 缺少验收标准对齐的用户故事
- 引用规约/计划中未定义的文件或组件的任务

#### D. 宪法对齐

- 与 MUST 原则冲突的任何需求或计划元素
- 缺少宪法规定的章节或质量门控

#### E. 覆盖差距

- 零相关任务的需求
- 没有映射需求/故事的任务
- 未反映在任务中的非功能需求（例如，性能、安全性）

#### F. 不一致

- 术语漂移（跨文件命名不同的相同概念）
- 计划中引用但规约中缺失的数据实体（反之亦然）
- 任务排序矛盾（例如，在没有依赖注释的情况下在基础设置任务之前的集成任务）
- 冲突需求（例如，一个需要 Next.js，另一个指定 Vue）

### 5. 严重性分配

使用此启发式优先级发现：

- **CRITICAL**：违反宪法 MUST、缺少核心规约文档或零覆盖阻止基线功能的需求
- **HIGH**：重复或冲突需求、模糊的安全/性能属性、不可测试的验收标准
- **MEDIUM**：术语漂移、缺少非功能任务覆盖、规约不足的边缘情况
- **LOW**：样式/措辞改进、不影响执行顺序的小冗余

### 6. 生成紧凑分析报告

输出 Markdown 报告（无文件写入），结构如下：

## 规约分析报告

| ID | 类别 | 严重性 | 位置 | 摘要 | 建议 |
|----|------|--------|------|------|------|
| A1 | 重复 | HIGH | spec.md:L120-134 | 两个相似需求... | 合并措辞；保留更清晰的版本 |

（每个发现一行；生成按类别首字母前缀的稳定 ID。）

**覆盖摘要表：**

| 需求键 | 有任务？ | 任务 ID | 注释 |
|--------|----------|---------|------|

**宪法对齐问题：**（如果有）

**未映射任务：**（如果有）

**指标：**

- 需求总数
- 任务总数
- 覆盖率 %（>=1 个任务的需求）
- 模糊计数
- 重复计数
- 关键问题计数

### 7. 提供下一步行动

在报告末尾，输出简洁的下一步行动块：

- 如果存在 CRITICAL 问题：建议在 `/speckit.implement` 之前解决
- 如果仅 LOW/MEDIUM：用户可以继续，但提供改进建议
- 提供明确的命令建议：例如，"运行 /speckit.specify 进行细化"、"运行 /speckit.plan 调整架构"、"手动编辑 tasks.md 以添加 'performance-metrics' 的覆盖"

### 8. 提供修复

询问用户："你想让我为前 N 个问题建议具体的修复编辑吗？"（不要自动应用它们。）

## 操作原则

### 上下文效率

- **最小高信号 token**：专注于可操作的发现，而不是详尽的文档
- **渐进式披露**：逐步加载文档；不要将所有内容转储到分析中
- **Token 高效输出**：将发现表限制为 50 行；总结溢出
- **确定性结果**：在没有更改的情况下重新运行应产生一致的 ID 和计数

### 分析指南

- **永不修改文件**（这是只读分析）
- **永不虚构缺失章节**（如果缺失，准确报告）
- **优先处理宪法违规**（这些始终是 CRITICAL）
- **使用示例而不是详尽规则**（引用特定实例，而不是通用模式）
- **优雅地报告零问题**（发出带有覆盖统计信息的成功报告）
